# CueMe — контекст проекта

> Рабочее название `CueMe` временное (есть конфликт по бренду — менять позже).
> Имя продукта держим в `config.py` (`APP_NAME`), чтобы смена названия была правкой одной строки.
> Не вшивай название в имена пакетов, классов и таблиц БД.

## Что строим

Telegram-бот: анализирует переписки пользователя (по его согласию, на его собственных
данных), строит модель **его** стиля общения и помогает писать сообщения под конкретного
собеседника. Ядро: «модель меня + наблюдения о собеседнике + рекомендация ответа».

## Позиционирование

Снаружи: «бот, который понимает, как ты общаешься, и помогает писать так, чтобы тебя
поняли правильно». Это инструмент самопонимания и коммуникации — НЕ «прочитай чужую
психику» и НЕ «продави нужный ответ». Никаких обещаний вроде «87% положительный ответ».

## Что делает — три слоя

1. **Модель себя** (фундамент, главный актив). Анализ собственного стиля юзера.
   Юридически чисто, технически надёжно (данных много, всё измеряется), уникально.
2. **Наблюдения о собеседнике** (осторожно). НЕ психологический профиль, а коммуникационные
   паттерны: «в этой переписке короткие сообщения с юмором заходят, длинные игнорятся».
   Подаётся как гипотезы по переписке, не как факты/диагноз.
3. **Ответ-ассистент** (killer-фича). Переформулирует черновик под регистр и тон
   собеседника, сохраняя собственный голос юзера.

## Архитектура (ключевой принцип — минимизировать вызовы LLM)

Пайплайн:
1. Загрузка JSON (экспорт чата из Telegram Desktop: `result.json`, поля `from`, `from_id`, `text`, `date`).
2. Парсер — делит сообщения по `from_id` на «я» и «контакт».
3. Локальные признаки по каждой стороне (БЕЗ LLM, детерминированно, бесплатно):
   средняя длина сообщений, латентность ответа, доля вопросов, эмодзи/пунктуация,
   баланс инициативы, регистр (ты/Вы, сленг). Это ~80% ценности.
4. Один LLM-проход НА ИМПОРТ → собирает `style_card` (голос юзера) и `interaction_card`
   (как эффективнее общаться с этим контактом). Кэшируется.
5. Хранилище.
6. Ответ-движок (дешёвая LLM) — короткий промпт: `[style_card] + [interaction_card] + черновик`.
7. Глубокий разбор (сильная LLM) — платная фича, апселл.

LLM зовётся только дважды: один раз на импорт + на каждый живой ответ дешёвой моделью.

## Стек

- Бот: **aiogram 3.x** (Python, polling на старте).
- Парсинг: stdlib `json`.
- Локальные признаки: чистый Python.
- Хранилище: **SQLite** на MVP → Postgres на рост. 4 таблицы: `users`,
  `contacts` (псевдоним `contact_id` вместо имени), `style_cards`, `interaction_cards`.
- LLM: один API, две модели (дешёвая на массовые ответы / сильная на глубокий разбор).
- Платежи: YooKassa (НЕ Telegram Stars — там ~30% комиссии).
- Хостинг: VPS.

## Юридические/этические ограничения (важно)

- Модель себя — чисто по 152-ФЗ (свои данные, своё согласие).
- О собеседнике хранить НЕ именованное досье, а производные признаки, привязанные к диалогу.
  Псевдонимизировать (`contact_id`). Формулировать как «как тебе эффективнее с ним писать»,
  а не «вот кто он».
- Модель себя = факты (надёжно). Наблюдения о собеседнике = гипотезы (осторожно).
  Эта разница зашита и в код, и в тексты для юзера.
- НЕ строим скрытый психологический профиль под «мягкими» формулировками. Предмет анализа
  честный: наблюдаемое коммуникационное поведение, не реконструкция психики.

## MVP (режем до вертикали)

Импорт одного чата → локальные признаки + одна карточка себя → фича «перепиши моё сообщение
под этот контакт». БЕЗ подписки, дашборда, глубокого разбора. Цель — снять короткое видео
и дать 10-20 людям потрогать.

## Соглашения по коду

- Чисто и минимально. Без переусложнений.
- Секреты только из `.env` через `config.py`. В код токены/ключи не вшивать.
- `APP_NAME` — единственное место с названием продукта.
